build-flow

An end-to-end pipeline that takes a human-approved design doc through plan → build → review → PR, fully autonomously, using multiple agent harnesses with fresh context per phase. The spec/[feature]/ directory is both the shared context and the durable state.

Overview

Today /spec produces an interactive, human-reviewed design.html and stops. build-flow picks up where it leaves off: once you verbally approve a design, a committed orchestrator script in ./bin drives the rest of the feature to a mergeable PR without supervision — generating a coding plan, getting it reviewed against the design, building the feature, validating it with typecheck/lint/tests/e2e, running an adversarial cross-harness code review with back-and-forth resolution, and finally opening and babysitting a PR that's up to date with main.

Two principles shape the whole design:

  1. The spec dir is a resumable state machine. Every phase is a pure function of "what's in spec/[feature]/ + the repo right now." All intermediate artifacts (plan, plan review, implementation notes, code-review rounds) are files in that directory. This makes the pipeline inspectable, resumable, and gives the design's two human-in-the-loop moments (approving the design; resolving an escalation) a natural file-based interface.
  2. Each phase runs as a fresh subprocess. "Fresh context between steps" and "different harness per step" both fall out of this for free: a new process is empty context, and switching harness is just switching which binary you invoke (claude vs codex vs gemini — all installed locally). Control flow — loops, gates, harness routing, escalation — lives in deterministic TypeScript, not in an agent's head.

The spec dir as state machine

spec/[feature]/
  design.html          ← INTERACTIVE output. Human-approved. The canonical target for everything downstream.
  state.json           ← orchestrator state: current phase, harness map, review round, status
  build-flow.log       ← append-only run log (what each phase did, timestamps)

  plan.md              ← coding plan                          (builder harness)
  plan-review.md       ← plan critiqued against design.html   (reviewer harness)
  implementation.md    ← what was built, divergences from plan (builder harness)
  review/
    round-1.md         ← reviewer findings + builder responses, in-file back-and-forth
    round-2.md
    ...

  NEEDS-INPUT.md       ← present ONLY when blocked on a human decision

state.json

{
  "feature": "build-flow",
  "phase": "review",            // plan | plan-review | build | validate | review | pr | monitor | done
  "status": "running",          // running | blocked | done | failed
  "reviewRound": 2,
  "branch": "amplified-geography",
  "harnessMap": {               // phase → harness binary + model
    "plan":        { "bin": "claude", "model": "opus" },
    "plan-review": { "bin": "codex" },
    "build":       { "bin": "claude", "model": "opus" },
    "review":      { "bin": "codex" },
    "pr":          { "bin": "claude", "model": "opus" }
  },
  "updatedAt": "2026-05-28T20:00:00Z"
}

Re-running build-flow reads state.json and continues from phase. There is no separate "resume" path — resuming is re-running, because all state is on disk.

How phases invoke harnesses

The orchestrator builds directly on the existing bin/ralph.ts pattern: spawn() the harness binary in --print/headless mode, stream stdout to build-flow.log, and watch for a completion sentinel. Each phase emits a structured verdict line the script parses to decide the transition.

Builder phases — claude --print 
spawn("claude", [
  "--print",
  "--dangerously-skip-permissions",   // headless, like ralph.ts
  "--model", harness.model,            // from harnessMap, e.g. "opus"
  buildPrompt(phase),                  // phase-specific instructions; cwd is the repo root,
])                                     // so the spec dir is already in scope — no --add-dir needed

Builder prompts reference the spec dir (by path) as the source of truth and end with an instruction to emit a sentinel (PLAN_DONE, BUILD_DONE, or ESCALATE: <reason>). The PR phase simply runs the existing skill: claude --print "/pr open". The same claude builder harness (the harnessMap.build entry) is reused wherever the builder side acts — build, validate-failure fixes, code-review responses, and monitor fixes.

Reviewer phases — codex exec 
spawn("codex", [
  "exec",
  reviewPrompt(phase),   // "Read design.html (canonical) + plan.md / the diff. Write the verdict file.
                         //  plan-review ends with: APPROVED | NEEDS_REVISION | ESCALATE
                         //  code-review  ends with: CLEAN | BLOCKING | ESCALATE"
])

Codex runs with a fresh context and no knowledge of how the builder reasoned — that independence is the point of the cross-harness split. codex exec review (the built-in review subcommand) is an option for phase 6; phases that must write a structured verdict file use a plain codex exec prompt.

Gate: plan-review before any code. Phase 3 is a hard gate — the reviewer judges plan.md strictly against design.html as the canonical target. The build phase never runs until the plan is APPROVED. The plan↔plan-review loop is unbounded (per your call), with ESCALATE available to either side.
Gate: deterministic validation. Phase 5 is run by the script, not an agent, so it can't be faked or hand-waved: bun run typecheck, bun run lint, bun run test, and e2e (see below). Any failure routes back to the builder with the captured failure output until green.

e2e & the dev server

e2e uses the existing browser tooling — the next-devtools MCP (Playwright under the hood) wired into the headless builder via --mcp-config. The standing "don't launch the dev server" rule exists to avoid duplicate dev-server processes in one worktree; build-flow relaxes it carefully with a launch-only-if-not-running guard owned by the script: 

// derive the dev URL exactly as CLAUDE.md documents
const devUrl = `https://${process.env.CONDUCTOR_WORKSPACE_NAME ?? basename(repoRoot)}.dispatch.localhost`
              // (CI/sandbox fallback: http://<name>.dispatch.localhost:${PORTLESS_PORT ?? 1355})

let devServer = null
if (!(await reachable(devUrl))) {
  devServer = spawn("bun", ["run", "dev"], { cwd: repoRoot })  // top-level, full toolchain
  await waitUntilReachable(devUrl)                              // poll readiness
}
// run e2e against devUrl (reused or freshly spawned) — happens either way
try {
  await runE2e(devUrl)
} finally {
  if (devServer) devServer.kill()  // tear down ONLY a server we started; never one you launched
}

The script probes the dev URL first; it spawns the top-level bun run dev (to get the full local toolchain) only when nothing is already serving, and it tears down only the process it started — never a dev server you launched yourself. This keeps the one-dev-server-per-worktree invariant intact while letting the autonomous run drive a real browser.

The code-review loop (phase 6)

Unbounded, file-based back-and-forth in spec/[feature]/review/:

  1. codex reviews the branch diff (git diff main...HEAD) against the design and writes review/round-N.md — each finding tagged [blocking] / [nit] / [question].
  2. If any [blocking] findings exist, claude reads round-N.md and responds in the same file, per finding: either fix (make the change, commit, note the SHA) or pushback (explain why it's intentional / wrong / out of scope). Fixes re-run the validate gate.
  3. The script re-invokes the reviewer for round-(N+1). The reviewer sees the prior round's resolutions and either accepts the pushbacks / confirms the fixes (CLEAN) or raises remaining/new blocking issues.
  4. Loop until the reviewer reports CLEAN.
Escalation valve. Both the reviewer and the builder are instructed: if you cannot converge — genuine disagreement, repeated thrash on the same point, or a question that needs product judgment — emit ESCALATE: <reason> instead of looping. The script then writes NEEDS-INPUT.md, rings the terminal bell, and halts. This is the "unbounded but with a human backstop" model you asked for: no round cap, but a clean way out when the agents are stuck.

PR creation & monitoring (phases 7–8)

Creation reuses the existing /pr open skill (rebase/merge main, push, open the PR) — no logic duplicated.

Monitoring is an explicit polling loop inside the script, per your preference — not the channels feature, not ScheduleWakeup tail-calls. GitHub has no general long-poll for PR events, so the loop polls on an interval:

while (true) {
  const pr = gh pr view --json state,mergeable,mergeStateStatus,statusCheckRollup
  const unresolved = gh api graphql (reviewThreads where !isResolved)

  if (state === MERGED || CLOSED) break
  if (mergeable === MERGEABLE && mergeStateStatus === CLEAN && unresolved === 0) break  // done

  if (mergeStateStatus === BEHIND)   → rebase onto main, push
  if (CI checks failing)             → dispatch claude builder with the failing logs
  if (unresolved > 0)                → dispatch claude builder with "/address-review <pr>"

  sleep(interval)   // e.g. 30–60s
}

The cloud pr-review.yml agent (Warp / GPT-5.5) is what produces the review threads on every push. Our monitor loop reacts to those threads by dispatching the local builder to address them — exactly the role /pr-monitor + /address-review play today, but driven by a deterministic loop instead of agent-scheduled wakeups. Thread resolution stays owned by the cloud review agent (per CLAUDE.md); the builder only fixes or pushes back.

Kickoff & status reporting

/spec stays design-only. The handoff:

  1. You verbally approve the design in the /spec conversation.
  2. A new /build-flow [feature] skill launches bun run bin/build-flow.ts [feature] as a background process and returns immediately.
  3. The script writes progress to state.json + build-flow.log. The skill (or you, any time) reads those to report status: current phase, review round, last log lines. When the script halts on a blocker, NEEDS-INPUT.md appears and the skill surfaces it.
  4. You resolve a blocker by editing the relevant artifact (or adding your decision to NEEDS-INPUT.md), deleting NEEDS-INPUT.md, and re-launching /build-flow [feature] — it resumes from state.json.
Because the heavy lifting is a headless OS process, the build survives indefinitely regardless of any Claude Code session's context budget — the session that launched it can end without killing the build. This is the key reason for the standalone-script host over in-session orchestration.

Substrate-agnostic by design

The spec dir is the contract and each phase is "run harness X with prompt Y over the spec dir." That keeps the execution substrate swappable: a phase could later be backed by Claude Code Workspaces, a cloud job, or a different CLI without touching the pipeline definition. We start with local claude --print / codex exec subprocesses because they're the simplest path to headless + multi-harness today.

Resolved decisions

  • e2e & dev server — next-devtools MCP (Playwright) via --mcp-config; script launches top-level bun run dev only if not already running, and tears down only what it started. (see "e2e & the dev server")
  • Worktree — out of scope. You kick off each run in a dedicated worktree, so build-flow operates on the current branch/worktree and does not manage worktrees itself.
  • Reviewer — codex only for now. Single reviewer; may add a second harness (e.g. gemini) later without changing the pipeline.
  • Notification — on halt (NEEDS-INPUT.md), the script emits a terminal bell (\a) and writes the blocker to the log; the /build-flow skill also surfaces it on next check.

Open Questions

  • Permission scoping. ralph.ts uses --dangerously-skip-permissions. For an autonomous build that writes code and runs git, is that acceptable, or do we want scoped --allowed-tools per phase (e.g. plan-review gets read-only)?
  • Cost / time ceiling. Unbounded review + validate loops have no hard cap. Do we want a soft budget (e.g. log a warning + bell past N rounds or M minutes) even though we're not auto-stopping?
